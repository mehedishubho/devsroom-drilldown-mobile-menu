## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PHP server → inline `<script>` (browser) | The `wp_add_inline_script` bridge crosses this boundary — a PHP value is serialized into a JS literal that the browser evaluates. `wp_json_encode` is the only sanctioned encoder. |
| Translation files (`languages/*.po` / `*.json`) → user-facing output | Translations are trusted (delivered by site admins / wordpress.org translation system). Standard WP trust model. The `.pot` itself ships no `msgstr` values — translators fill them in `.po` copies. |
| Plugin header → WordPress loader | The `Domain Path:` header is read by WP at activation to locate `languages/`. No user input reaches it. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|

The single value (`'noResults' => __('No results', ...)`) is a static string literal under developer control, not user input. `wp_json_encode` escapes quotes, slashes, and unicode line-separators per RFC 8259.
