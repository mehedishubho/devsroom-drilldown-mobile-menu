       a. `$items = wp_get_nav_menu_items( $menu );`
       b. Guard: `if ( ! $items || empty( $items ) ) { return []; }`
       c. **Pass 1** — Index by ID. Declare `$indexed = [];`. Foreach over `$items as $item`:
          ```php
          $indexed[ (int) $item->ID ] = [
              'id'           => (int) $item->ID,
              'title'        => $item->title,
              'url'          => $item->url,
              'target'       => $item->target ?? '',
              'classes'      => is_array( $item->classes ) ? $item->classes : [],
              'has_children' => false,
              'children'     => [],
          ];
          ```
          (The `is_array` guard defends against WP returning a string classes value in edge cases; per D-04 classes is always an array in output.)
       d. **Pass 2** — Attach children to parents. Declare `$root_ids = [];`. Foreach over `$items as $item`:
          ```php
          $parent_id = (int) $item->menu_item_parent;

          if ( $parent_id > 0 && isset( $indexed[ $parent_id ] ) ) {
              $indexed[ $parent_id ]['children'][]   = & $indexed[ (int) $item->ID ];
              $indexed[ $parent_id ]['has_children'] = true;
          } else {
              $root_ids[] = (int) $item->ID;
          }
          ```
          NOTE: The `& $indexed[ (int) $item->ID ]` is a localized reference assignment inside a VALUE-based foreach (not `foreach &$ref`). This is safe per RESEARCH.md A3 — the reference is resolved before return and no reference variable persists in scope. (If you prefer zero references, use the recursive closure alternative from RESEARCH.md Pattern 1 "Anti-references alternative" — both produce identical output. Choose one; the reference-minimal version above is shorter and recommended.)
       e. **Pass 3** — Extract roots. Declare `$tree = [];`. Foreach over `$root_ids as $id`:
          ```php
          $tree[] = $indexed[ $id ];
          ```
       f. `return $tree;`
    6. Close the method and class. Add a class-level docblock: "Converts flat wp_get_nav_menu_items() output into a nested tree. Pure data — no HTML, no panel IDs, no Elementor dependency."

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WP Admin (menu author) → wp_get_nav_menu_items() | Menu items are created by trusted WP admins. The tree builder reads this trusted data and produces pure data structures. No user-facing output crosses this boundary in Phase 2. |
| WpNavTree::build() → caller (Plan 01 render / Phase 4 renderer) | The builder returns arrays/strings only. The downstream renderer (Phase 4) is the trust boundary where escaping is enforced. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Tampering | Tree builder data flow | accept (deferred) | The tree builder outputs PURE DATA (arrays of strings). It performs no HTML output and no URL redirection. It does NOT escape because escaping is the renderer's responsibility (D-03, ASVS V7). Menu titles/URLs are trusted admin-authored data. Phase 4 must apply `esc_html()` to titles and `esc_url()` to URLs at render time. No mitigation action in Phase 2 — by design. |
| T-02-06 | Tampering (WooCommerce endpoint URLs) | WooCommerce menu items | accept | Per D-06, WooCommerce registers Cart/My Account/Checkout/Shop as standard nav menu items. `wp_get_nav_menu_items()` resolves their URLs via WP core (same code path as all other items). The tree builder treats them identically — no special-case branch that could introduce a divergence. WooCommerce URL correctness verification is Phase 7 (COMP-03). Not a Phase 2 threat. |
| T-02-07 | Denial of Service (deeply nested menu) | Recursive tree resolution | accept | WordPress nav menus have no enforced depth limit, but realistic menus are ≤5 levels deep. The 3-pass algorithm is O(n) in Pass 1 and Pass 2 and O(roots) in Pass 3 — linear time, no recursion depth risk in the recommended reference-minimal implementation. Even the recursive alternative uses PHP closures with depth bounded by menu nesting (typically <10). No DoS mitigation required; menu size is bounded by practical admin usage. |

**No `high` severity threats in this phase.** The tree builder produces pure data and outputs nothing to the browser. All HTML/URL security is deferred to Phase 4's renderer, which will own the `esc_html()`/`esc_url()` trust boundary. This matches the locked decision D-03 and the ASVS V7 posture documented in RESEARCH.md Security Domain.
</threat_model>
