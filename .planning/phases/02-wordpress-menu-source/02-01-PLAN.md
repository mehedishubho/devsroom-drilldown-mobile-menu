       /**
        * Build SELECT options from all registered WordPress nav menus.
        *
        * @return array<int|string, string> Menu term_id => menu name. Includes empty default.
        */
       protected function get_wp_menu_options(): array {
           $menus   = wp_get_nav_menus();
           $options = [ '' => esc_html__( '— Select a Menu —', 'devsroom-drilldown-mobile-menu' ) ];

           if ( empty( $menus ) ) {
               return $options;
           }

           foreach ( $menus as $menu ) {
               $options[ $menu->term_id ] = $menu->name;
           }

           return $options;
       }

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WP Admin (menu author) → wp_options (menu data) | Menu items are created by WP admins (trusted role). Tree builder reads trusted data. |
| Elementor editor → widget render() | `wp_menu_id` value is constrained by SELECT control to real `term_id`s; Elementor serializes the option keys at control-registration time. User cannot inject an arbitrary menu ID via the editor UI. |
| render() → browser output | The ONLY new HTML output in Phase 2 is the `.ddmm-editor-hint` div with a hardcoded translated string. No dynamic/user data is echoed in this phase. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | `wp_menu_id` setting (render) | mitigate | `wp_menu_id` originates from a SELECT control whose `options` are built server-side from real `WP_Term->term_id` values. Elementor validates settings against registered control options on save. If a non-existent term_id is passed, `wp_get_nav_menu_items()` returns false and `WpNavTree::build()` returns `[]` → empty-state hint shows. No injection vector. |
| T-02-02 | Information Disclosure | Editor hint HTML output | mitigate | The hint text is a hardcoded `esc_html__()` translated literal ("Select a menu to display"). No user/DB data is interpolated into the output. `esc_html__` escapes for HTML context, neutralizing any XSS (though none is possible since the string is a literal). |
| T-02-03 | Spoofing (Stored XSS via menu title) | Tree data flow | accept (deferred) | Menu titles flow through `WpNavTree::build()` as raw trusted admin data. Phase 2 does NOT output titles as HTML — the tree is pure data (D-03). Phase 4's renderer is responsible for `esc_html()`/`esc_url()` at output time. Not a Phase 2 threat because no HTML is rendered from tree data in this phase. |
| T-02-04 | Spoofing (Open redirect via menu URL) | Tree data flow | accept (deferred) | Menu URLs come from `wp_get_nav_menu_items()` (WP core resolves them, admin-trusted). Phase 2 stores them as pure data; no links are output. Phase 4 must use `esc_url()`. Not a Phase 2 threat. |

**No `high` severity threats in this phase.** The only HTML output is a hardcoded, escaped, translated editor hint. All dynamic menu data is deferred to Phase 4 rendering, where escaping will be enforced.
</threat_model>
