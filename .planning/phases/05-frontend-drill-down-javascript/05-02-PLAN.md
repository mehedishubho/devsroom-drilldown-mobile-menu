<contracts>
<!-- The search-box DOM contract. From 05-CONTEXT.md D-06..D-11 and 05-RESEARCH.md §Search Box Markup. -->

The search box MUST emit this structure (hook attributes are the hard contract — JS in Plan 04 queries them):

```html
<div class="ddmm-search" data-ddmm-search role="search">
    <label class="screen-reader-text" for="ddmm-search-input-{widget_id}">{translatable 'Search menu items'}</label>
    <input type="search"
        id="ddmm-search-input-{widget_id}"
        class="ddmm-search__input"
        data-ddmm-search-input
        placeholder="{search_placeholder, default 'Search menu…'}"
        autocomplete="off"
        aria-controls="ddmm-search-results-{widget_id}">
    <ul class="ddmm-search__results"
        data-ddmm-search-results
        id="ddmm-search-results-{widget_id}"
        aria-live="polite"
        aria-relevant="additions"></ul>
</div>
```

Hook attributes (hard contract):
- `data-ddmm-search` on the outer wrapper (Plan 04 may use to scope search to this widget instance)
- `data-ddmm-search-input` on the `<input>` (Plan 04 binds the debounced input listener here)
- `data-ddmm-search-results` on the `<ul>` (Plan 04 writes filtered results here)

ID uniqueness: append `-{widget_id}` to input and results IDs so multiple widget instances on the same page do not collide (label `for` / `aria-controls` / `aria-controls` all resolve correctly per instance). `$widget_id` is already a render() parameter.

Placement: Called from render() AFTER `self::render_header( $settings )` and BEFORE the `<nav>` printf, gated by `! empty( $settings['search_enabled'] ) && 'yes' === $settings['search_enabled']`.

    ```php
    /**
     * Render the search box (Phase 5 D-10).
     *
     * Emits a sticky bar with a search input and an empty results container.
     * The input carries data-ddmm-search-input; the results ul carries
     * data-ddmm-search-results. JS (ddmm-frontend.js) owns the live filtering.
     *
     * The container is EMPTY on render — JS populates results via DOM APIs
     * (document.createElement + textContent, never innerHTML with user input —
     * ASVS V5). The "No results" message is also emitted by JS so it is
     * translatable via the same text domain.
     *
     * @param array  $settings  Widget settings (search_placeholder).
     * @param string $widget_id Elementor widget ID (for unique input/results IDs).
     * @return void Echos HTML directly.
     */
    private static function render_search_box( array $settings, string $widget_id ): void {
        $placeholder = ! empty( $settings['search_placeholder'] )
            ? $settings['search_placeholder']
            : __( 'Search menu…', 'devsroom-drilldown-mobile-menu' );

        printf(
            '<div class="ddmm-search" data-ddmm-search role="search">' .
                '<label class="screen-reader-text" for="ddmm-search-input-%1$s">%2$s</label>' .
                '<input type="search" id="ddmm-search-input-%1$s" class="ddmm-search__input" data-ddmm-search-input placeholder="%3$s" autocomplete="off" aria-controls="ddmm-search-results-%1$s">' .
                '<ul class="ddmm-search__results" data-ddmm-search-results id="ddmm-search-results-%1$s" aria-live="polite" aria-relevant="additions"></ul>' .
            '</div>',
            esc_attr( $widget_id ),
            esc_attr__( 'Search menu items', 'devsroom-drilldown-mobile-menu' ),
            esc_attr( $placeholder )
        );
    }
    ```

    - `%1$s` reuses `$widget_id` four times (label for, input id, aria-controls, results id) so the same widget instance has matching IDs.
    - The input has NO value attribute (empty on render).
    - The `<ul>` is empty on render (no `<li>` children).
    - The placeholder defaults to 'Search menu…' when the setting is empty (D-09 + Claude's Discretion on default copy).
    - All three values pass through `esc_attr` / `esc_attr__` (the placeholder is user-configured free text — MUST be escaped per ASVS V5 / PLUG-06).
    - The text domain is `devsroom-drilldown-mobile-menu` (not abbreviated).

## Trust Boundaries

| Boundary | Description |

| Elementor settings → rendered search box HTML | The `search_placeholder` TEXT control value crosses from the editor into the rendered `placeholder` attribute |
| Rendered HTML → JS filtering pipeline (next plan) | The empty results container is the contract surface; Plan 04 populates it with user-typed query results |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | Tampering / XSS | `search_placeholder` value emitted into `placeholder` attribute | mitigate | Pass through `esc_attr()` in the printf. The placeholder is the only user-controlled free-text value in the search box markup. `esc_attr` encodes quotes and angle brackets, preventing attribute breakout. Low risk. |
| T-05-06 | Tampering / XSS (FUTURE) | User-typed search query reflected into results `<ul>` by Plan 04 JS | mitigate (in Plan 04) | This plan emits an EMPTY `<ul>` on render — no user query flows through PHP. The XSS surface is entirely in Plan 04's JS-side rendering, which MUST use `document.createElement` + `.textContent` (never `innerHTML` with the raw query). Documented in 05-RESEARCH.md §Security Domain. This plan does not introduce the surface; it is noted here for cross-plan traceability. |
| T-05-07 | Information disclosure | Empty results container visible before user types | accept | The empty `<ul>` renders no content (no `<li>` children); `aria-live="polite"` only announces when content is added. No information leaked. |

**ASVS V5 compliance:** Only one free-text value (`search_placeholder`) reaches HTML; it is escaped via `esc_attr()`. All other values are static strings or the `$widget_id` (Elementor-controlled, also `esc_attr`-escaped). The user-typed search query is handled entirely by JS in Plan 04, which has its own threat model covering the DOM-API rendering requirement.
